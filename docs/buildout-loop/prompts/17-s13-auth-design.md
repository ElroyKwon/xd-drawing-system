# S13 — 실 로그인 + 매직링크 초대 + 외부 업로드 [설계/스캐폴드 · 프로덕션 인증=HUMAN_GATE]

> 신규 스코프. 밤샘 자율 범위: **설계 + 명확히 게이트된 mock 스캐폴드까지**. 실 자격증명·세션 보안·프로덕션 인증 도입은 **HUMAN_GATE**(LOOP.md·GATE-6). 버전추적은 **S3에서 이미 구현**(파일/폴더 CRUD + 명시적 버전세트) — S13은 그 위에 **실 인증만** 접붙이는 설계.

## 현 상태 (S7 로컬 모의)
- `auth.py`: `require_role`이 `store.get_current_user()`(서버 전역 가변 member_id)로 역할 강제. 세션·비밀번호·토큰·매직링크 **전무**. `PUT /api/auth/me`로 누구나 아무 member로 전환(보안 경계 아님, GATE-3에서 "표시용" 하향).
- 즉 RBAC 로직(require_role, 역할 모델, 프로젝트-구성원)은 **실동작**하나, "누가 요청자인가"의 **인증**만 모의.

## 목표 (실 인증 도입 시)
1. **실 로그인**: 요청자 신원을 세션 토큰으로 확립(전역 가변 → 요청별 인증 컨텍스트).
2. **매직링크 초대**: 관리자가 이메일로 초대 → 수신자가 링크 클릭 → 세션 확립(비밀번호리스). **S11 이메일 인프라 재사용**.
3. **외부 업로드**: 초대된 외부 사용자가 링크로 접속 → 제한 스코프 세션 → 특정 폴더 업로드 → **버전추적(S3) 그대로 적용**.

## 설계

### 인증 모델
- **세션**: 서명·만료 토큰(예: HMAC 서명 + exp) 쿠키 또는 Authorization 헤더. `get_current_user()`를 **요청 컨텍스트에서 토큰 검증**으로 교체(전역 상태 제거 → GATE-3 프라이버시 경계 실현).
- **패스워드리스 우선**: 매직링크(이메일 1회용 토큰) → 검증 시 세션 발급. 비밀번호 저장 회피(자격증명 관리 부담↓).

### 매직링크 플로우 (S11 이메일 재사용)
```
관리자 → POST /api/auth/invite {email, project, role}
      → 서명·만료 초대토큰 생성 → email_service.send_email(매직링크 URL 포함)
수신자 → GET /api/auth/magic?token=... → 토큰 검증(서명·만료·1회성)
      → 세션 발급(쿠키) → 프로젝트에 role로 구성원 추가
```

### 외부 업로드 플로우
```
외부자(매직링크 세션) → 제한 스코프(특정 프로젝트/폴더, 업로드 전용 역할)
      → POST /api/drawings(업로드) → 변환 → 버전세트 편입(S3: is_latest·이력·보관)
```
버전추적 로직은 신규 아님 — S3 `add_version`/버전세트가 그대로 처리. 실 인증이 "누가 올렸나"를 실제 신원으로 채움.

### 마이그레이션 (S7 → 실 인증)
- `require_role`·역할모델·project_member는 **보존**(이미 실동작). 바뀌는 것은 `get_current_user()` 소스뿐: 전역 → 세션 토큰.
- `PUT /api/auth/me`(모의 전환)는 **개발 전용**으로 격하 또는 실 인증 활성 시 비활성.

## 게이트된 mock 스캐폴드 (프로덕션 아님 — 데모용)
> 실 세션 보안 없이 **플로우만** 시연. `XD_AUTH_REAL=1` 없으면 세션 미확립(S7 모의 유지).
- `POST /api/auth/invite`(mock): 서명 토큰 생성 + **S11 mock 이메일로 매직링크 URL을 outbox에 기록**(실 발송 0). 실제 세션·구성원 추가는 게이트 뒤.
- `GET /api/auth/magic?token=`(mock): 토큰 서명·만료 검증만 시연, 실 세션 쿠키는 게이트.
- 명확 표기: 이 스캐폴드는 **인증 경계가 아님**(GATE-6 전).

## HUMAN_GATE (GATE-6 — 프로덕션 인증)
사용자 결정 필요:
1. **인증 방식**: 매직링크 단독 / 비밀번호 병행 / SSO(OAuth·SAML) / 사내 IdP.
2. **세션 보안**: 토큰 서명키 관리·만료·회전·쿠키 보안속성(HttpOnly/Secure/SameSite), CSRF.
3. **외부 노출**: 외부 사용자 접속 = 인터넷 노출 → 배포·HTTPS·레이트리밋·감사. (LOOP.md "외부 배포"·"클라우드 백엔드 전환" 게이트 연동.)
4. **이메일 egress**: 매직링크 발송 = 실 이메일 → GATE-5 의존.
5. **개인정보**: 실 사용자 이메일·신원 저장 → 보관·삭제 정책.

## Acceptance (설계 스테이지)
- **R1** 현 S7 모의의 정확한 한계와 실 인증 목표를 문서화(위).
- **R2** 매직링크·외부업로드·마이그레이션 설계가 기존 자산(S3 버전추적·S7 RBAC·S11 이메일) 재사용을 명시.
- **R3** 프로덕션 인증 결정사항을 GATE-6로 HUMAN_GATE에 등재.
- **R4** (선택) 게이트된 mock 스캐폴드는 실 세션을 확립하지 않음(플로우 시연만)을 코드/문서로 명확화.

## Out of scope (게이트 전 금지)
- 실 세션·쿠키·토큰 보안 구현 · 비밀번호 저장 · SSO 연동 · 외부 인터넷 노출/배포 · 실 매직링크 이메일 발송(GATE-5). **전부 GATE-6 이후.**
